# 🧠 Agent #1 — Transaction Pattern Analyzer
_Sentinel (Agent ThinkING)_

The **Transaction Pattern Analyzer** is the **first agent** in Sentinel’s fraud pipeline.  
Its single responsibility is to analyze an **individual transaction** against global & contextual patterns (amount, merchant, location, time, velocity) and emit a **risk signal** with **clear flags** and **deterministic reasoning**.

> This agent **does not take actions**. It only **scores & explains**.  
> Actions are performed later by the **Action Executor Agent**.

---

## ✨ What this agent outputs
- `risk_score` (0–100) — normalized composite score
- `flags` — e.g., `AMOUNT_SPIKE`, `GEO_MISMATCH`, `UNUSUAL_TIME`, `UNUSUAL_VELOCITY`, `RARE_MCC`
- `reasoning` — short human‑readable explanation (deterministic)
- `feature_contributions` — numeric contributors (e.g., z‑scores, distances)
- `version` — agent/config version for audit

---

## 🧱 Step‑by‑Step Plan

### 1) Define the contract (ADK‑friendly JSON)
**Request** (minimum viable context so this agent can run first in the chain):
```json
{
  "transaction_id": "tx_123",
  "customer_id": "cust_456",
  "amount": 249.99,
  "currency": "EUR",
  "merchant_id": "m_789",
  "merchant_category": "ELECTRONICS",
  "channel": "CARD_PRESENT",
  "geo": { "country": "NL", "city": "Amsterdam", "lat": 52.37, "lon": 4.90 },
  "timestamp_utc": "2026-02-24T14:03:00Z",
  "reference_context": {
    "global_avg_by_mcc": 58.2,
    "global_stdev_by_mcc": 41.0,
    "recent_velocity_last_1h": 0,
    "recent_velocity_last_24h": 4,
    "last_known_geo": { "lat": 52.09, "lon": 5.12 },
    "last_tx_timestamp_utc": "2026-02-24T12:40:00Z"
  }
}
````

**Response**

```json
{
  "risk_score": 86,
  "flags": ["AMOUNT_SPIKE", "GEO_MISMATCH", "UNUSUAL_TIME"],
  "reasoning": "High amount (z=3.1), unusual hour (02:13 UTC), travel 520km since last txn 1h ago.",
  "feature_contributions": {
    "amount_zscore": 3.1,
    "geo_distance_km": 520.0,
    "velocity_1h": 3
  },
  "version": "pattern-v1.0.0",
  "config_version": "weights-2026-02-24"
}
```

***

### 2) Implement signals (pure functions, return \[0..1] + optional flag)

*   **Amount Spike**: `z = (amount − global_avg_mcc) / stdev_mcc` → map via sigmoid to \[0..1]; flag `AMOUNT_SPIKE`.
*   **Geo Mismatch**: Haversine distance from `last_known_geo`; scale to \[0..1]; if large distance in short time ⇒ flag `GEO_MISMATCH`.
*   **Velocity**: compare `recent_velocity_last_1h/24h` to MCC baselines; flag `UNUSUAL_VELOCITY`.
*   **Rare MCC / Merchant Rarity**: category/merchant rarity vs baselines; flag `RARE_MCC`.
*   **Unusual Time**: hour‑of‑day anomaly for the MCC; flag `UNUSUAL_TIME`.

> Start rule‑based. Keep each signal **stateless & testable**. Add ML later.

***

### 3) Blend into a single score

*   Normalize each signal to **\[0..1]**
*   Weighted sum (from config), e.g.:
    *   `amount=0.35, geo=0.25, velocity=0.20, mcc=0.10, time=0.10`
*   `risk_score = 100 * Σ(weight_i * signal_i)` → clamp \[0..100]

Return **top contributors** (z‑score, distance, velocity counts) for transparency.

***

### 4) Compose deterministic reasoning

Generate a **short, factual sentence** from the triggered flags & contributors.  
Keep this LLM‑free by default (deterministic & auditable).  
Later, you can optionally paraphrase with a constrained LLM while preserving facts.

***

### 5) Expose the HTTP API

    POST /score/pattern    # body = Request JSON, returns Response JSON
    GET  /healthz

*   Validate schema; return 400 on invalid payload.
*   Include `version` & `config_version` in every response.

***

### 6) Configuration & policy

*   `config/weights.yaml` — per‑signal weights
*   `config/thresholds.yaml` — per‑signal/flag thresholds
*   Version these files; include `config_version` in logs & responses.

***

### 7) Observability & audit

*   **Structured logs**: txnId, score, flags, contributors, duration\_ms, version, config\_version
*   **Metrics**: score histogram, p95 latency, error rate
*   **Tracing**: correlation ID from the Orchestrator

***

### 8) Testing strategy

*   **Unit tests**: amount z‑score, geo distance, velocity, rare MCC, time window
*   **Golden tests**: canonical LOW/MED/HIGH examples → stable outputs
*   **Property tests**: monotonicity (↑ z‑score ⇒ never ↓ risk\_score)
*   **Contract tests**: request/response schema validation

***

### 9) Performance targets

*   p95 **< 50–100 ms**
*   Stateless service, horizontally scalable

***

### 10) Future enhancements (optional)

*   MCC‑specific seasonality & holiday effects
*   Clustering / isolation forest for pattern novelty
*   Device/IP/merchant reputation enrichers
*   Constrained LLM paraphrase for analyst‑friendly text (facts unchanged)

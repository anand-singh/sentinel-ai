# 🧾 Agent #3 — Evidence Builder (Explainability Agent)
_Sentinel (Agent ThinkING)_

The **Evidence Builder** combines outputs from upstream agents and produces a **clear, auditable, human‑readable explanation** for why a transaction looks risky.  
It **does not score or take actions**. It aggregates, normalizes, and explains.

> Inputs: outputs from Pattern Analyzer (Agent #1), Behavioral Risk (Agent #2), and (optionally) AML/Compliance.  
> Outputs: a structured **evidence bundle** used by the Aggregated Risk Scorer (Agent #4) and the Dashboard.

---

## ✨ What this agent returns

- `evidence_summary` — short, deterministic explanation sentence(s)
- `combined_flags` — deduplicated list across agents, ordered by severity/impact
- `agent_scores` — per‑agent scores for transparency
- `explanation_items` — traceable details per source agent
- `version` — agent/config version for audit

---

## 🧱 Step‑by‑Step Plan

### 1) Define the contract (ADK‑friendly JSON)

**Request (input)**
```json
{
  "transaction_id": "tx_123",
  "customer_id": "cust_456",

  "pattern_agent_output": {
    "risk_score": 72,
    "flags": ["AMOUNT_SPIKE","GEO_MISMATCH"],
    "reasoning": "Amount unusually high; location far from prior point.",
    "version": "pattern-v1.0.0"
  },

  "behavioral_agent_output": {
    "behavioral_risk_score": 64,
    "flags": ["NEW_DEVICE","UNUSUAL_TIME"],
    "feature_contributions": {
      "amount_zscore_customer": 3.2,
      "geo_distance_km": 520.5
    },
    "reasoning": "New device; amount far above customer's norm; off-hours.",
    "version": "behavior-v1.0.0"
  },

  "aml_agent_output": {
    "aml_score": 55,
    "flags": ["PEP_MATCH_PENDING"],
    "reasoning": "Hit on PEP screening pending analyst verification.",
    "version": "aml-v1.0.0"
  }
}
````

**Response (output)**

```json
{
  "evidence_summary": "High amount vs pattern, new device off-hours, abnormal location distance; AML check pending.",
  "combined_flags": ["AMOUNT_SPIKE","NEW_DEVICE","UNUSUAL_TIME","GEO_MISMATCH","PEP_MATCH_PENDING"],
  "agent_scores": {
    "pattern_score": 72,
    "behavioral_score": 64,
    "aml_score": 55
  },
  "explanation_items": [
    { "source": "pattern_agent", "details": "Amount spike; geo mismatch." },
    { "source": "behavioral_agent", "details": "New device; off-hours; amount 3.2σ above norm." },
    { "source": "aml_agent", "details": "PEP pending." }
  ],
  "version": "evidence-v1.0.0",
  "config_version": "evidence-policy-2026-02-24"
}
```

***

### 2) Merge logic (deterministic & auditable)

*   **Combine flags**
    *   Deduplicate (`Set`), sort by **priority → frequency → recency**.
    *   Priority rules come from `config/evidence-policy.yaml`.

*   **Compose summary**
    *   Template‑based sentence builder.
    *   Pull only **facts** from inputs (no hallucination).
    *   Prefer concise phrasing with max N tokens (configurable).

*   **Attach scores & items**
    *   Include per‑agent scores for transparency.
    *   Include `explanation_items` mapping each statement to a source agent.

*   **Versioning**
    *   Return `version` (agent) and `config_version` (policy/templates).

> Keep the Evidence Builder **LLM‑free** by default for determinism.  
> (Optional) Add a **constrained paraphrase** step later using Vertex AI (low temperature, schema guardrails) without changing facts.

***

### 3) Evidence policy & templates

*   `config/evidence-policy.yaml`
    *   **flag\_order**: e.g., `CRITICAL` > `HIGH` > `MED` > `LOW`
    *   **flag\_priority\_map**: `{ AMOUNT_SPIKE: HIGH, NEW_DEVICE: HIGH, ... }`
    *   **summary\_templates**:
        *   With AML: `"High amount vs pattern, {behavioral_phrase}, {geo_phrase}; AML check pending."`
        *   Without AML: `"High amount vs pattern, {behavioral_phrase}, {geo_phrase}."`

Keep templates **short**, **factual**, and **slot‑based**.

***

### 4) HTTP API

    POST /evidence/build     # body = Request JSON → returns Response JSON
    GET  /healthz

*   Validate payload strictly (JSON schema).
*   Reject unknown fields; return helpful 400 errors.
*   Include `version` & `config_version` in every response.

***

### 5) Observability & audit

*   **Structured logs**: `txnId`, `combined_flags`, `summary`, `agentsPresent`, `duration_ms`, `version`, `config_version`
*   **Metrics**: summary length histogram, p95 latency, error rate, missing‑agent rate
*   **Tracing**: correlation ID propagated from Orchestrator

***

### 6) Testing strategy

*   **Unit**
    *   `mergeFlags()` dedup & sort
    *   `composeSummary()` correctness with/without AML
    *   `buildEvidencePayload()` end‑to‑end

*   **Golden**
    *   Canonical cases: *pattern‑only*, *pattern+behavior*, *with AML* → stable outputs

*   **Negative/robustness**
    *   Missing agent sections; unknown flags; empty arrays

*   **Contract**
    *   JSON schema validation for request/response

***

### 7) Performance targets

*   p95 **< 30–60 ms** (string ops + simple merges only)
*   Stateless; horizontally scalable

***

### 8) Future enhancements (optional)

*   Multi‑language summaries via template catalogs
*   Per‑segment templates (SME, retail, VIP)
*   Evidence richness level (short/long) toggle for UI
*   Optional Gemini paraphrase with **strict factuality guard**

## 🔒 Security notes

*   Treat inputs as **already sanitized** by upstream agents, but re‑validate schemas.
*   Mask PII in logs; store only correlation IDs.
*   Include `version` & `config_version` in logs/responses for auditability.
*   Principle of least privilege when deployed (IAM).

## ☁️ Deploy (Cloud Run example)

```bash

  jib:build -Djib.to.image=eu.gcr.io/$GCP_PROJECT_ID/evidence-builder:$(git rev-parse --short HEAD)

# Deploy
gcloud run deploy evidence-builder \
  --image eu.gcr.io/$GCP_PROJECT_ID/evidence-builder:$(git rev-parse --short HEAD) \
  --region $GCP_REGION --allow-unauthenticated=false \
  --set-env-vars "APP_ENV=prod,CONFIG_PATH=/app/config"
```

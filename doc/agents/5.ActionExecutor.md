# 🛡️ Agent #5 — Action Executor (ADK‑style)
_Sentinel (Agent ThinkING)_

The **Action Executor** is the final step in the Sentinel pipeline.  
It receives **final risk decisions** from the **Aggregated Risk Scorer (Agent #4)** and executes **only approved, policy‑governed tools** (e.g., freeze transaction, notify security, create case report, step‑up auth, escalate to human).  
It **does not score**, **does not explain**, and **does not improvise**. It runs **explicitly registered tools** under strict policy control (ADK principles).

> Think of this agent as the **safe hands**: deterministic, auditable, and minimal in scope.

---

## ✨ Outputs (What this agent returns)

- `executed_actions` — ordered list of tools executed for the transaction
- `status` — `SUCCESS | PARTIAL_SUCCESS | FAILED`
- `timestamp_utc`, `audit_id` — for traceability
- `version` — agent version  
- **(always) deterministic & fully auditable**

---

## 🔗 Upstream dependency

- **Agent #4 — Aggregated Risk Scorer** provides:
  - `final_risk_score` (0–100)
  - `severity` (`LOW | MED | HIGH | CRITICAL`)
  - `recommended_action` (`ALLOW | REVIEW | CHALLENGE | BLOCK`)
  - `explanation` (for context only; not used to decide tools)

Actions are mapped **by policy**, not by free‑form text.

---

## 🧱 Step‑by‑Step Plan

### 1) Define the contract (ADK‑friendly JSON)

**Request (input)**
```json
{
  "transaction_id": "tx_123",
  "customer_id": "cust_456",
  "final_risk_score": 87,
  "severity": "CRITICAL",
  "recommended_action": "BLOCK",
  "explanation": "Amount spike + new device + geo mismatch; AML moderate.",
  "policy_version": "policy-2026-02-24",
  "correlation_id": "8c6c-...-f1b2"
}
````

**Response (output)**

```json
{
  "executed_actions": ["freeze_transaction","notify_security_team","create_case_report"],
  "status": "SUCCESS",
  "timestamp_utc": "2026-02-24T14:40:00Z",
  "audit_id": "audit_99900221",
  "version": "executor-v1.0.0"
}
```

> The executor **must not** add or invent actions. It only runs tools allowed by policy for the given `severity` / `recommended_action`.

***

### 2) Register the action tools (explicit ADK‑style)

Provide **clear function contracts** and register each tool:

*   `freeze_transaction(transaction_id)`  
    Freezes/blocks the transaction in the core system.

*   `notify_security_team(transaction_id, summary)`  
    Sends a structured alert (with correlation ID) to SOC/analyst queue.

*   `create_case_report(details)`  
    Creates a case artifact (summary, flags, scores, evidence link) in Case Store.

*   `request_step_up_auth(customer_id)`  
    Triggers step‑up authentication (e.g., OTP, app approval) when risk is high but not yet critical.

*   `escalate_to_human(transaction_id)`  
    Assigns the case to a human analyst or escalated queue.

> Tools should be **idempotent** (safe to retry) and produce **structured results** for audit.

***

### 3) Map policy → actions (no logic in code)

Define mappings in **YAML** so risk ops can adjust without redeploys:

`config/action-policy.yaml`

```yaml
version: "action-policy-2026-02-24"
mappings:
  LOW:
    ALLOW: []
  MED:
    REVIEW: [create_case_report]
  HIGH:
    CHALLENGE: [request_step_up_auth, create_case_report]
  CRITICAL:
    BLOCK: [freeze_transaction, notify_security_team, create_case_report]
fallbacks:
  default: [create_case_report]
```

Rules:

*   If `(severity, recommended_action)` is missing, use `fallbacks.default`.
*   The executor **must** log the policy version applied.

***

### 4) Safety & governance guardrails

*   **No free‑form actions** — only tools from the registered catalog.
*   **Deterministic execution order** — as listed in the policy.
*   **Timeouts & retries** — bounded retries; if a tool fails, record failure and continue if policy allows partial success.
*   **Audit line per tool call** — includes `transaction_id`, `tool_name`, `args_masked`, `result_status`, `latency_ms`, `policy_version`, `agent_version`, `correlation_id`.
*   **Idempotency keys** — to avoid duplicate effects on retries.
*   **PII handling** — mask sensitive fields in logs; store only required references/IDs.

***

### 5) HTTP API

    POST /action/execute      # body = Request JSON from Aggregated Risk Scorer
    GET  /healthz

*   Validate schema strictly; 400 on invalid payloads.
*   Return full result with `executed_actions`, `status`, `audit_id`, `timestamp_utc`, and `version`.

***

### 6) Observability & audit

*   **Structured logs**: correlation ID, policy & agent versions, actions executed, per‑tool status, duration.
*   **Metrics**: actions per severity, tool success/failure rate, p95 latency, retry counts.
*   **Tracing**: propagate correlation IDs from Orchestrator for end‑to‑end visibility.

***

### 7) Testing strategy

*   **Policy mapping tests** — each `(severity, recommended_action)` path.
*   **Tool execution tests** — mock each tool; ensure correct order & error handling.
*   **Negative tests** — unknown recommendation, missing fields, tool failure → `PARTIAL_SUCCESS`/`FAILED`.
*   **Contract tests** — request/response schema validation.
*   **Idempotency tests** — repeated calls with same `audit_id` or idempotency key do not duplicate effects.

***

### 8) Performance targets

*   p95 **< 50–100 ms** for policy→action mapping + orchestration (not including external system latency).
*   Tools should target **sub‑second** completion; long‑running flows should be asynchronous with status callbacks or job IDs.

***

### 9) Deployment & permissions

*   **Runtime**: Cloud Run (recommended)
*   **Permissions**: least‑privilege IAM; separate service accounts for:
    *   Executor service
    *   Downstream core banking APIs
    *   Case store writer
*   **Networking**: private access to core systems; no public endpoints unless behind gateway.

***

## 🔒 Security notes

*   Reject unknown fields and invalid schemas.
*   Mask PII in logs; carry only correlation IDs.
*   Include `policy_version` & `version` in all logs and responses.
*   Ensure tool endpoints are **private** and protected with IAM or mTLS.
*   Record idempotency keys to avoid duplicate effects.

***

## ☁️ Deploy (Cloud Run example)

```bash
# Build container with Jib
mvn -pl agents/action-executor-java -DskipTests package \
  jib:build -Djib.to.image=eu.gcr.io/$GCP_PROJECT_ID/action-executor:$(git rev-parse --short HEAD)

# Deploy
gcloud run deploy action-executor \
  --image eu.gcr.io/$GCP_PROJECT_ID/action-executor:$(git rev-parse --short HEAD) \
  --region $GCP_REGION --allow-unauthenticated=false \
  --set-env-vars "APP_ENV=prod,ACTION_POLICY_FILE=/app/config/action-policy.yaml"
```

# 🧩 Agent #2 — Behavioral Risk Agent
_Sentinel (Agent ThinkING)_

The **Behavioral Risk Agent** scores **how unusual a transaction is for a specific customer**.  
It compares the incoming transaction with that customer’s historical behavior (amount ranges, active hours, geo patterns, device/IP, merchant mix, velocity) and emits a **behavioral risk score** with **clear flags** and **deterministic reasoning**.

> This agent **does not execute actions**. It only **scores & explains**.  
> Actions are performed by **Agent #5 – Action Executor** after Agent #4 aggregates scores.

---

## ✨ Outputs (What this agent returns)

- `behavioral_risk_score` (0–100) — normalized composite score
- `flags` — e.g., `NEW_DEVICE`, `NEW_IP_RANGE`, `UNUSUAL_TIME`, `GEO_DEVIATION`, `RARE_MERCHANT`, `BURST_ACTIVITY`
- `feature_contributions` — numeric contributors (e.g., amount z‑score vs _customer_, geo distance/velocity, novelty booleans)
- `reasoning` — short, human‑readable, deterministic summary
- `version` — agent/config version for audit

## 🧱 Step‑by‑Step Plan

**Response (output)**

```json
{
  "behavioral_risk_score": 77,
  "flags": ["NEW_DEVICE","UNUSUAL_TIME","GEO_DEVIATION"],
  "feature_contributions": {
    "amount_zscore_customer": 3.2,
```

> Keep the **profile snapshot immutable** per request (no live DB reads in the scoring path unless cached).  
> The profile is maintained upstream (e.g., Dataflow jobs, nightly batch, or a feature store).

***

### 2) Implement behavioral signals (pure functions → \[0..1] + optional flag)

*   **Amount vs Customer Baseline**  
    `z = (amount − avg_amount) / stdev_amount` → map via sigmoid to \[0..1]; flag `AMOUNT_DEVIATION` if |z| exceeds threshold.
*   **Unusual Time**  
    Check if `timestamp_utc` is inside `sleep_hours_utc` → high score; near edges → medium; otherwise low; flag `UNUSUAL_TIME`.
*   **Geo Deviation / Impossible Travel**  
    Haversine distance + time delta from `last_geo`; compute implied speed; flag `GEO_DEVIATION` if unrealistic.
*   **New Device Fingerprint**  
    If not in `usual_devices` → score 1.0; decay after N repeats; flag `NEW_DEVICE`.
*   **New IP Range**  
    If IP CIDR not in `usual_ip_ranges` → flag `NEW_IP_RANGE`.
*   **Merchant/MCC Novelty**  
    If MCC weight is low in `usual_mcc_distribution` or merchant not in `usual_merchants_top` → flag `RARE_MERCHANT`/`RARE_MCC`.
*   **Burst/Velocity vs Personal Baseline**  
    If count in recent window >> `velocity_baseline_per_hour` → flag `BURST_ACTIVITY`.
*   **Channel Shift**  
    New/uncommon channel vs `usual_channels` → flag `CHANNEL_SHIFT`.

> Keep each signal **stateless & testable**. These feed a blender that computes one score.

***

### 3) Blend signals → one behavioral score

*   Normalize each signal to **\[0..1]**
*   Use weights from `config/behavior-weights.yaml` (example):
    *   `amount=0.25, geo=0.25, device=0.20, time=0.10, merchant=0.10, velocity=0.10`
*   `behavioral_risk_score = 100 * Σ(weight_i * signal_i)` → clamp \[0..100]

Return **top contributors** to support explainability.

***

### 4) Compose deterministic reasoning

Assemble a concise sentence from triggered flags + top contributors (no LLM required):  
`"Amount 3.2σ above normal, new device, off‑hours, travel 520km since last txn 1h ago."`  
You can later add a constrained LLM (low temperature, schema guardrails) purely for **paraphrasing**.

***

### 5) Expose the HTTP API

    POST /score/behavior     # body = Request JSON; returns Response JSON
    GET  /healthz

*   Validate schema strictly; return 400 on invalid payloads.
*   Include `version` and `config_version` in every response.

***

### 6) Configuration & policy

*   `config/behavior-weights.yaml` — per‑signal weights
*   `config/behavior-thresholds.yaml` — thresholds per flag
*   Version files; put `config_version` into logs & responses.

***

### 7) Observability & audit

*   **Structured logs**: txnId, customerId (hashed), score, flags, contributors, duration\_ms, version, config\_version
*   **Metrics**: score histogram, p95 latency, error rate
*   **Tracing**: correlation ID from Orchestrator

***

### 8) Testing strategy

*   **Unit tests**: each signal function + blender
*   **Golden tests**: canonical LOW / MED / HIGH scenarios
*   **Property tests**: monotonicity (↑ amount z‑score ⇒ ↑ behavioral score, others fixed)
*   **Contract tests**: request/response schema validation

***

### 9) Performance targets

*   p95 **< 50–100 ms**
*   Stateless; horizontal scaling

***

### 10) Future enhancements

*   Seasonality & travel‑mode heuristics (vacation / business trip windows)
*   Reputation features (device/IP/merchant), decay functions
*   Feature store integration & drift monitoring
*   Optional Gemini (Vertex AI) paraphrase with guardrails (facts unchanged)
